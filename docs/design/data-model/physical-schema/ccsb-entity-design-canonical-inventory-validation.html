CCSB Entity Design and Canonical Inventory Validation

This report defines the six missing Dataverse tables needed to close the V1 control, publish, audit, lock, capability, and outbound reliability gaps: ccsb_resourcecapability, ccsb_operationaudit, ccsb_schedulechange, ccsb_publishsnapshotitem, ccsb_publishlock, and ccsb_outboxevent.

Design stance: add these as an additive schema baseline, keep all delete behavior Restrict, make existing-table lookups metadata-optional for upgrade safety, and enforce business-required links in validation and Custom APIs.

1. Design Principles

  • Keep canonical logical names stable: no renames of existing CCSB tables or fields in a minor release.
  • Use explicit tables for audit, lock, change, snapshot item, outbox, and capability records instead of burying V1 control behavior in large JSON fields.
  • Use Restrict delete on all relationships so evidence cannot disappear through parent deletion.
  • Use user-owned tables where board/business-unit scoped security matters; use organization-owned tables for immutable publish evidence and system queue records.
  • Use Dataverse auditing for platform-level history and ccsb_operationaudit for product-level reason, policy, correlation, and field-scope evidence.
  • Reserve outbound integration through ccsb_outboxevent; do not enable connector dispatch until an explicit integration capability is approved.

Controlled ledger of proposed, committed, published, superseded, or rolled-back schedule mutations.

Dynamics alignment: Maps to changes that would otherwise be made directly to bookings, assignments, or requirements in Field Service/Project Operations. CCSB keeps the ledger product-owned so all mutations flow through Custom APIs.

4. Logical-Name Validation Test Plan

Automate this in CI: run the validation against dbmodel.xlsx, provisioner schema JSON, unpacked solution XML, and live Dataverse metadata inventory before every schema PR and before every managed solution export.

Entity regex: ^ccsb_[a-z][a-z0-9]*$

Field regex: ^ccsb_[a-z][a-z0-9]*$

Relationship regex: ^(ccsb_[a-z0-9]+|systemuser)_(ccsb_[a-z0-9]+)(_[a-z0-9]+)?$

Required automated checks

  1. Every custom CCSB table logical name starts with ccsb_ and is lowercase.
  2. Every proposed table has primary id ccsb_<entity>id and primary name ccsb_name.
  3. Every custom field starts with ccsb_, except Dataverse system columns.
  4. Every lookup field ends with id and its target table exists in current or proposed metadata.
  5. Every new relationship uses Delete Restrict unless an explicit architecture decision approves otherwise.
  6. Alternate key fields are Text, Whole Number, Decimal, DateTime, Choice, or Lookup; are not field-secured; and stay below Dataverse key size limits.
  7. New fields on existing tables are optional at metadata level, then enforced by validation at activation/publish time.
  8. No existing logical name is renamed or dropped in a minor release.
  9. Payload, image, diagnostic, reason, and actor display fields are either minimized or field-secured.
  10. Evidence/control tables have table audit enabled and are protected by server-side append-only/update rules.

Backward compatibility checks

  1. Additive tables and optional columns are minor-version compatible.
  2. Changing type, logical name, ownership, or key composition is major-version breaking.
  3. Changing choice integer values is breaking; adding new choice values is minor-version compatible when consumers tolerate unknown values.
  4. Tightening a required field at metadata level is breaking for existing environments; enforce requiredness through activation validation first.

Upgrade path recommendation

  1. v1.1 Additive schema: create the six tables, relationships, alternate keys, field security profiles, and audit settings. Keep all new lookups optional at metadata level.
  2. v1.1 Backfill mode: create resource capabilities and optional snapshot items from trustworthy existing JSON/manifests only. Record all backfills as ccsb_operation plus ccsb_operationaudit.
  3. v1.1 Compatibility gate: validation warns when publish lock, snapshot item, schedule change, or rollback field flags are absent for active boards.
  4. v1.2 Enforcement: publish/rollback Custom APIs require schedulechange, publishlock, snapshotitem, and operationaudit evidence for new publishes.
  5. v2.0 Breaking boundary, only if needed: remove legacy manifest-only or JSON-only compatibility after customer environments pass migration evidence gates.

6. Design Decisions and Alignment Notes

Field Service and URS alignment

Resource capabilities mirror bookable resource characteristics, categories, territories, proficiency/rating, resource type, work-hour availability, and enable-for-availability concepts without requiring CCSB to depend on Field Service tables.

Project Operations alignment

Roles, requirements, and capabilities stay generic enough to model project/resource skills and roles, while CCSB remains server-authoritative through Custom APIs rather than direct client mutation.

Generic Dataverse alignment

Alternate keys use supported non-secured columns; all relationship deletes are Restrict; payload/reason/error fields use field security; auditing is enabled at environment/table/column levels.

Publish and rollback alignment

Snapshot item and schedule change records provide field-scoped rollback evidence. Publish locks prevent official-state mutation without approved override and audit evidence.
