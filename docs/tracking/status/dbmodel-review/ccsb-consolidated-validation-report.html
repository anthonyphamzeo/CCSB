Alignment Strengths

  • The schema covers the main V1 configuration backbone: ccsb_board, ccsb_boardversion, entity/field/relationship mappings, group definitions, status models/transitions, resource roles, rule definitions, permission profiles/assignments, configuration validation results, and runtime projection.
  • The native schedule graph is partially represented through ccsb_scheduleitem, ccsb_scheduleassignment, and ccsb_schedulerequirement. Schedule item includes item type, planned/requested dates, source entity/record references, publish state, parent item, service, location, board, board version, and schedule version.
  • Operation governance is partially modeled: ccsb_operation has correlation ID, idempotency key, type/status, payload/result JSON, timing, retry, retention, and item counts; ccsb_operationitem has item keys, source/target record references, correlation, payloads, result payloads, retry state, and error fields.
  • Publish is partially modeled with ccsb_publishsnapshot containing scope, status, snapshot number, state hash, manifest JSON, payload pointer, operation lookup, predecessor snapshot, retention date, record count, and size.
  • Security intent is visible in ccsb_permissionprofile and ccsb_permissionassignment, and the foundation docs correctly state these permissions must sit below Dataverse security rather than bypass it.

What Works

  • The UI catalogs are internally coherent: every business field in the audited Core schema is placed exactly once on a Main form, all views are intentionally narrow, and related navigation uses restrictive delete behavior.
  • The forms follow sensible governance: General, Advanced, Lifecycle, and Administration tabs separate daily fields from JSON/payload/error details and lifecycle evidence.
  • The design correctly avoids Quick Create for operational/evidence records such as schedule items, assignments, board versions, snapshots, conflicts, projections, and operations.
  • The UI package preserves existing base forms/views by creating Planora-named forms and views rather than overwriting defaults.

Improvement Recommendations

Database and Architecture

  1. Close the schema-control gap before runtime work: add or explicitly replace schedulechange, publishsnapshotitem, publishlock, auditlog/operationaudit, and reserved outboxevent designs.
  2. Make audit strategy executable: choose Dataverse table auditing, custom immutable audit records, or both. Capture actor, timestamp, correlation, reason, operation, result, affected records, and retention policy.
  3. Normalize or formally govern activity/capability shortcuts: document whether ccsb_scheduleitem is the canonical activity table and whether resource capability lives in JSON or a normalized child table.
  4. Propagate correlation: add ccsb_correlationid and operation references across conflict, publish, rollback, lock, change, snapshot item, audit, and outbox records.
  5. Harden sensitive fields: field-secure payload JSON, result JSON, error detail, resolver references, integration references, source record identifiers, and advanced configuration fields.
  6. Review keys and query shapes: prove alternate keys for board code, version number/label, semantic mapping keys, role/status keys, projection scope/content hash, operation correlation/idempotency, and conflict deduplication keys.

UI/UX

  1. Keep the current forms/views as admin scaffolding: do not mistake them for the runtime workspace.
  2. Create app-level navigation: build role-specific model-driven app modules/sitemaps once schema and security roles settle.
  3. Add high-value subgrids selectively: Board versions/configurations, Schedule requirements/assignments, Operation items, Status definitions/transitions, Calendar rules/exceptions, and publish evidence.
  4. Design the PCF workspace around the documented workflows: board selector, virtualized resource tree/timeline, details panel, workbenches, validation warning/block states, publish preview, lock/rollback state, and keyboard/form alternatives.

Backlog Governance

  1. Update backlog status from evidence: if the schema and UI deliverables count as completed or partially completed work, mark corresponding tasks with evidence links. Otherwise leave them Not Started and plan implementation accordingly.
  2. Split US-004 into hard gates: physical schema approval, Dataverse import proof, migration strategy, security matrix, data factory, and automated schema validation.
  3. Do not pull deferred stories into V1: mobile/offline, connectors, split/merge, URS adapters, and autonomous AI remain design gates.
