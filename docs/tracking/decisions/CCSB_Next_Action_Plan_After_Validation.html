CCSB Next Action Plan After Validation

The CCSB direction remains sound, but the current bulk Model-Driven UX provisioner is not ready for further full-scale execution. Authentication and metadata discovery now work, yet generated FormXML and SavedQuery LayoutXML have repeatedly violated Dataverse contracts. Because the same generator is used across 39 tables and 78 views, this is a shared release-pipeline defect rather than an Availability Window-only issue.

Recommended decision: pause the 39-table / 78-view rerun. First complete read-only environment inventory, controlled cleanup planning, schema-governance decisions, and a five-table UX pilot in DEV. Resume batch generation only after the pilot passes browser, security, export, and clean-import gates.

2. Immediate Recovery: Stabilize and Inventory First

P0 Preserve evidence and stop duplicated component creation.

  1. Freeze the bulk provisioner. Do not run another full 39-table pass just to test one XML change.
  2. Export a backup of the current DEV state. Export the unmanaged ccsb_modeldrivenuix solution and record version, publisher, solution ID, and component counts.
  3. Preserve execution evidence. Save the console log, source ZIP/hash, generated FormXML/LayoutXML/FetchXML, actual ObjectTypeCode, and failure timestamp.
  4. Use a disposable or recoverable DEV environment. Do not test repair versions in UAT/production or in a non-restorable DEV environment.
  5. Build one read-only discovery tool. It must make zero metadata changes and generate JSON/CSV/HTML inventory outputs.

3. Controlled Cleanup and Recovery

Do not delete the existing solution or generated components blindly. The live run created at least some forms before it failed at view creation.

Preferred path — targeted cleanup

  • Identify only Planora – * forms/views owned by ccsb_modeldrivenuix.
  • Generate a dry-run report with component ID, name, table, component type, solution, and intended action.
  • Require explicit -ConfirmCleanup for deletion.
  • Delete only failed generated artifacts; never touch base forms/views, tables, or customer customizations.
  • Recreate from a proven pilot template, not from unverified generic XML.

Fallback — clean DEV environment

  • Use this if ownership is ambiguous, duplicates exist, or cleanup cannot be proven safe.
  • Import Core and Foundation Schema into a clean DEV environment.
  • Run discovery first, then the five-table pilot only.
  • Export the clean UX solution as the ALM baseline.
Do not change the solution unique name merely to bypass cleanup. That creates a permanent ALM duplication problem. Preserve ccsb_modeldrivenuix as the intended solution identity; use controlled cleanup or a clean environment when necessary.

5. Correct UX Automation Strategy

Do not continue with broad, hand-authored XML dialects. Dataverse UI XML is strict and target-environment-dependent. The next tool should be template-driven and fail before it creates components.

Form strategy

  1. Retrieve valid target-environment Main and Quick Create forms as baseline templates.
  2. Clone only approved structural nodes and allowed attributes.
  3. Bind fields only after live metadata confirms the field exists and has a supported control type.
  4. Keep record metadata such as form type, name, ID, and activation state out of FormXML.
  5. Validate generated FormXML locally against the approved template contract before Dataverse create/update.

View strategy

  1. Retrieve a valid public view baseline for the same table or proven category.
  2. Set grid/@object from the live numeric ObjectTypeCode; never from logical name and never omit it.
  3. Build FetchXML and LayoutXML together: layout row ID must be fetched; every visible cell must map to a returned field.
  4. Validate object code, query type, row ID, cell order, widths, and primary ID inclusion before sending the request.
  5. Create Current and All Records views only after the pilot table passes in browser and app designer.
Boundary: the Model-Driven package is administration/support scaffolding. The schedule board runtime remains a separate React PCF + Custom API stream, with its own accessibility, command, and concurrency work.

9. Risks and Explicit “Do Not Do” Decisions

Risks to manage

  • Repeated partial runs can accumulate duplicate forms/views and create unusable maker experience.
  • Schema changes after UX scale-out will cause significant rework and inconsistent user-facing terminology.
  • Payload/error/configuration fields can be exposed through generic forms without field security and role testing.
  • Static source checks do not prove ALM portability; only clean export/import evidence does.
  • App modules and sitemaps remain fragile until actual table names, UX components, and security role behavior are stable.

Do not do these next

  • Do not run another broad generator build solely to test grid/@object.
  • Do not hand-assemble solution XML for forms, views, app modules, sitemaps, or web resources.
  • Do not change solution unique names just to bypass partial components.
  • Do not mark US-004 complete without audit, migration, security, and clean import/export evidence.
  • Do not treat Model-Driven forms as the runtime Schedule Board.
Recommended next owner action: build the read-only inventory utility and run a schema hardening workshop first. Then implement the five-table UX pilot, not another full rerun.
